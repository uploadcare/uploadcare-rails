# Migrating from uploadcare-rails 3.x to 4.0

This guide covers upgrading from uploadcare-rails 3.x (jQuery widget) to 4.0 (File Uploader, Web Components-based).

## Requirements

**Before upgrading**, ensure your project meets the new minimum requirements:

| Dependency | 3.x | 4.0 |
|---|---|---|
| Ruby | 2.7+ | 3.3+ |
| Rails | 6.0+ | 7.2+ |

If you're on an older Ruby or Rails version, stay on the 3.x line:

```ruby
gem "uploadcare-rails", "~> 3.4"
```

## Step 1: Update the gem

```ruby
# Gemfile
gem "uploadcare-rails", "~> 4.0"
```

```console
$ bundle update uploadcare-rails
```

## Step 2: Update the configuration

Re-run the config generator to get the new template:

```console
$ rails g uploadcare_config
```

The generator will prompt before overwriting an existing file. You can also compare the new template with your current initializer and update manually. The key changes:

### Renamed options

| 3.x (Widget) | 4.0 (File Uploader) | Notes |
|---|---|---|
| `images_only` | `img_only` | Renamed to match File Uploader API |
| `tabs` | `source_list` | Value format changed (see below) |
| `input_accept_types` | `accept` | Standard accept string |
| `preferred_types` | `external_sources_preferred_types` | |
| `multipart_min_size` | `multipart_min_file_size` | |
| `preview_proxy` | `secure_delivery_proxy` | |
| `cdn_base` | `cdn_cname` | |
| `camera_mirror_default` | `camera_mirror` | |
| `crop` | `crop_preset` | |

### Storage behavior: `do_not_store` is not renamed to `store`

**No action needed** — `do_not_store` still works the same way for server-side model callbacks. The new `store` option is a separate client-side setting and is not a drop-in replacement:

- `do_not_store` — server-side, controls `mount_uploadcare_file` / `mount_uploadcare_file_group` callbacks
- `store` — client-side, controls File Uploader behavior (`<uc-config store="...">`)

```ruby
# 4.0 — keep your existing do_not_store setting as-is
config.do_not_store = true

# Optional, separate client-side setting
config.store = "false"
```

### Removed options (no equivalent in File Uploader)

These options were specific to the jQuery widget and have no direct replacement:

- `live` — Web Components initialize automatically, no manual start needed
- `manual_start` — same as above
- `preview_step` — File Uploader handles preview differently
- `clearable` — handled by the File Uploader UI natively
- `system_dialog` — not applicable to Web Components
- `audio_bits_per_second` — not supported
- `video_preferred_mime_types` — not supported
- `video_bits_per_second` — not supported
- `locale_translations` — File Uploader uses a different localization system
- `locale_pluralize` — same as above

### Value format changes

**Source list** (formerly "tabs"):

The value is now a comma-separated string instead of an array. Source names have also changed:

| 3.x source | 4.0 source |
|---|---|
| `file` | `local` |
| `url` | `url` |
| `camera` | `camera` |
| `facebook` | `facebook` |
| `gdrive` | `gdrive` |
| `dropbox` | `dropbox` |
| `instagram` | `instagram` |
| `evernote` | *(removed)* |
| `flickr` | *(removed)* |
| `onedrive` | `onedrive` |

See the [File Uploader source list docs](https://uploadcare.com/docs/file-uploader/configuration/#source-list) for the full list of available sources.

```ruby
# 3.x
config.tabs = %w[url file facebook]

# 4.0
config.source_list = "local,url,facebook"
```

**Accept types**:

```ruby
# 3.x
config.input_accept_types = %w[image/*]

# 4.0
config.accept = "image/*,video/*,.pdf"
```

### Async job defaults changed

In the generated config template, async jobs now default to `false`:

```ruby
# 3.x template default
config.store_files_async = true
config.delete_files_async = true

# 4.0 template default
config.store_files_async = false
config.delete_files_async = false
```

**Warning:** If you re-run the config generator and were relying on async store/delete (e.g., with Sidekiq), the new defaults will cause files to be stored/deleted synchronously in the request cycle. This may cause slower requests or timeouts. If you use async jobs, explicitly set these to `true` in your initializer after regenerating the config:

```ruby
config.store_files_async = true
config.delete_files_async = true
```

Async jobs require a configured ActiveJob adapter (e.g., Sidekiq).

## Step 3: Review `uploadcare_include_tag` usage

### No-argument usage (no template change)

If your layout already has a plain include tag, the call site stays the same:

```erb
<%# 3.x and 4.0 %>
<%= uploadcare_include_tag %>
```

No template change is required for this case, but the rendered assets/output are different in 4.0:

| | 3.x Output | 4.0 Output |
|---|---|---|
| JS | `<script src="https://ucarecdn.com/libs/widget/3.x/uploadcare.full.min.js">` | `<script type="module">import * as UC from '...file-uploader.min.js'` |
| CSS | none (bundled in JS) | `<link rel="stylesheet" href="...uc-file-uploader-regular.min.css">` |
| Config | `<script>UPLOADCARE_PUBLIC_KEY = '...';</script>` | Configured per-component via `<uc-config>` |

### Parameter migration (if you passed options)

```erb
<%# 3.x %>
<%= uploadcare_include_tag(version: '3.x', bundle: 'full', min: true) %>

<%# 4.0 %>
<%= uploadcare_include_tag(version: 'v1', solution: 'regular', min: true) %>
```

- `bundle:` is no longer accepted. Ruby will raise `ArgumentError` if passed.
- `solution:` replaces it. Valid values: `"regular"`, `"inline"`, `"minimal"`.

### Alternative: Importmap setup (Rails 7+)

For modern Rails apps using importmaps:

```console
$ rails g uploadcare_importmap
```

Then in your layout, use `uploadcare_stylesheet_tag` instead of the full include tag:

```erb
<%= uploadcare_stylesheet_tag %>
```

## Step 4: Update uploader fields in views

This is the most impactful change. The old jQuery widget used hidden `<input>` elements; the new File Uploader uses Web Components.

### Basic field helper

```erb
<%# 3.x %>
<%= uploadcare_uploader_field :post, :picture %>
<%# Output: <input role="uploadcare-uploader" type="hidden" name="post[picture]" ...> %>

<%# 4.0 %>
<%= uploadcare_uploader_field :post, :picture %>
<%# Output: <uc-form-input> + <uc-config> + <uc-file-uploader-regular> + <uc-upload-ctx-provider> %>
```

The helper name is the same, but the **method signature changed**:

```ruby
# 3.x — positional options hash
uploadcare_uploader_field(:post, :picture, { multiple: true })

# 4.0 — keyword arguments only
uploadcare_uploader_field(:post, :picture, multiple: true)
```

If you were passing a positional Hash as the third argument, you must switch to keyword arguments.

### Standalone field tag

```erb
<%# 3.x %>
<%= uploadcare_uploader_field_tag :file, value: "..." %>

<%# 4.0 %>
<%= uploadcare_uploader_field_tag :file %>
```

The `value:` parameter is no longer supported. The `<uc-form-input>` Web Component manages its own hidden input value based on uploaded files. For edit forms, see [Pre-populating fields on edit](#pre-populating-fields-on-edit) below.

### Multiple file uploads (file groups)

The `multiple` attribute is still auto-detected from `mount_uploadcare_file_group`, same as in 3.x. No changes needed for basic usage:

```erb
<%# This still works — multiple is auto-detected from mount_uploadcare_file_group %>
<%= uploadcare_uploader_field :post, :attachments %>
<%= f.uploadcare_file :attachments %>
```

You can also set `multiple` explicitly to override the auto-detected value:

```erb
<%= uploadcare_uploader_field :post, :attachments, multiple: false %>
```

### Pre-populating fields on edit

In 3.x, the hidden `<input>` could be pre-populated via the `value:` attribute. In 4.0, the `<uc-form-input>` Web Component manages its own hidden input — there is no `value:` parameter on the helpers.

For edit forms, the File Uploader does not display previously uploaded files. The uploaded file URL is submitted with the form as before, but showing a preview of the existing file is your responsibility. A common pattern:

```erb
<%= form_with model: @post do |f| %>
  <% if @post.picture.present? %>
    <p>Current: <%= image_tag @post.picture.transform_url(preview: { dimensions: "200x200" }) %></p>
  <% end %>
  <%= f.uploadcare_file :picture %>
  <%= f.submit %>
<% end %>
```

### FormBuilder integration (new in 4.0)

4.0 adds FormBuilder support (which was removed back in 2.0). You can now use `uploadcare_file` inside `form_with` / `form_for`:

```erb
<%= form_with model: @post do |f| %>
  <%= f.uploadcare_file :picture %>
  <%= f.uploadcare_file :attachments, multiple: true, group_output: true %>
  <%= f.submit %>
<% end %>
```

The `group_output: true` option tells the File Uploader to submit a single file group UUID (e.g., `uuid~N`) instead of separate input fields for each file. Use it with `mount_uploadcare_file_group` attributes so the value matches the expected group URL format.

This is the recommended approach for new code.

## Step 5: Update custom CSS and JavaScript

### CSS selectors

All old widget CSS classes are gone. Update any custom styles:

| 3.x Selector | Replacement |
|---|---|
| `.uploadcare--widget` | `uc-file-uploader-regular` (custom element) |
| `.uploadcare--dialog` | The File Uploader handles its own modal |
| `[role="uploadcare-uploader"]` | `uc-form-input` |
| `.uploadcare--widget_status_ready` | No direct equivalent; use File Uploader events |

### JavaScript

The old widget exposed a global `uploadcare` object and `UPLOADCARE_*` variables. These are gone.

```javascript
// 3.x — REMOVE
UPLOADCARE_PUBLIC_KEY = "your_key";
var widget = uploadcare.Widget('[role=uploadcare-uploader]');
widget.onChange(function(file) { ... });

// 4.0 — use Web Component events instead
// See https://uploadcare.com/docs/file-uploader/events/ for the full event reference
const ctxProvider = document.querySelector('uc-upload-ctx-provider');
ctxProvider.addEventListener('file-upload-success', (e) => {
  const fileInfo = e.detail;
  // handle uploaded file
});
ctxProvider.addEventListener('file-removed', (e) => {
  // handle file removal
});
```

### Turbolinks / Hotwire workarounds

The old README included workarounds for Turbolinks/Hotwire caching issues that removed `.uploadcare--widget` and `.uploadcare--dialog` elements on navigation. **These workarounds are no longer needed** and should be removed. Web Components handle lifecycle automatically.

```javascript
// 3.x — REMOVE THIS ENTIRE BLOCK
document.addEventListener('turbo:before-cache', function() {
    const widgets = document.querySelectorAll('.uploadcare--widget');
    widgets.forEach(widget => { widget.remove(); });
});
```

## Step 6: Verify models and configuration code (mostly unchanged)

The model layer (`mount_uploadcare_file`, `mount_uploadcare_file_group`) and all REST/Upload API interfaces are **unchanged**. No model code changes are required.

The server-side behavior (storing files after save, deleting files after destroy, caching, API calls) works exactly as before.

If you reference configuration internals in custom code, note these deprecations:

| 3.x | 4.0 | Action |
|---|---|---|
| `Uploadcare::Rails.configuration.uploader_parameters` | `Uploadcare::Rails.configuration.uploader_config_attributes` | Returns a Hash of Web Component attributes instead of JS global variable assignments |
| `Uploadcare::Rails.configuration.widget` | `Uploadcare::Rails.configuration.uploader_config_attributes` | Same as above |
| `Uploadcare::Rails::Configuration::WIDGET_PARAMS` | `Uploadcare::Rails::Configuration::FILE_UPLOADER_PARAMS` | Constant listing recognized config keys |

The old methods and constants still exist but are deprecated and will be removed in a future version.

## Step 7: Update tests

If you have tests that assert on the HTML output of uploader helpers, update them:

```ruby
# 3.x assertions — REMOVE
expect(html).to include('role="uploadcare-uploader"')
expect(html).to include('type="hidden"')
expect(html).to include('data-multiple="true"')

# 4.0 assertions — ADD
# Use substring matching; the exact attributes and order may vary
expect(html).to include('<uc-form-input')
expect(html).to include('<uc-config')
expect(html).to include('<uc-file-uploader-regular')
expect(html).to include('<uc-upload-ctx-provider')
```
